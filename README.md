# Produced by Lucid Website

Next.js marketing site with a self-hosted TinaCMS admin embedded in the same app.

## Local development

1. Install dependencies:

```bash
npm install
```

2. Copy `.env.example` to `.env.local`.

For local development, only `NEXT_PUBLIC_SITE_URL` is really needed.
The `dev` script already forces `TINA_PUBLIC_IS_LOCAL=true`, so GitHub and KV credentials are not required locally.

3. Start the app:

```bash
npm run dev
```

4. Open:

- Site: `http://localhost:3000`
- CMS: `http://localhost:3000/admin`

In local mode, Tina runs against the local filesystem and does not require the GitHub/KV backend.

If Tina says the datalayer server is already using port `9000`, stop the existing `tinacms dev` or Node process first and rerun `npm run dev`.

## Tina structure in this repo

This project does not use the same file layout as the Tina tutorial starter, and that is okay.

- `tina/config.ts` is the source of truth for the Tina schema and collections.
- `tina/__generated__/` is auto-generated by Tina and will be rebuilt for you.
- `pages/api/tina/[...routes].ts` exposes the local/self-hosted Tina API.
- `content/` contains the editable JSON and Markdown files.

The tutorial's `collections/`, `fields/`, and `queries/` folders are just one way to organize a Tina config. In this repo, everything is consolidated into `tina/config.ts`.

`config.ts` also does not need to be `config.tsx` unless you are embedding React-based Tina UI components directly in the config.

## Production environment variables

For production editing on Vercel, set these variables:

- `NEXT_PUBLIC_SITE_URL`
- `NEXTAUTH_URL`
- `NEXTAUTH_SECRET`
- `GITHUB_OWNER`
- `GITHUB_REPO`
- `GITHUB_PERSONAL_ACCESS_TOKEN`
- `GITHUB_BRANCH`
- `KV_REST_API_URL`
- `KV_REST_API_TOKEN`

If those variables are missing, the site still builds, but Tina falls back to local mode and production content editing will not persist correctly on Vercel.

## Build

```bash
npm run build
```

This runs:

- `node scripts/tina-build.mjs`
- `next build`

The Tina build script is environment-aware:

- Local and fallback builds use `tinacms build --skip-indexing`
- Self-hosted production builds run full Tina indexing so existing GitHub content is written into the Tina KV index

If production skips indexing, the live Tina dashboard can load with empty collections even though the site content exists in `content/`.
